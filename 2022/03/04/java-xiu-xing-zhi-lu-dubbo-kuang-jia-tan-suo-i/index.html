Dubbo框架探索I

dubbo的工作原理？dubbo的注册中心挂了是否能继续通信？一次rpc调用的流程？

工作原理：

第一层：service层，接口层，给服务提供者和消费者来实现的（provider和consumer）

第二层：config层，配置层，主要是对dubbo进行各种配置的（提供配置文件进行参数配置）

第三层：proxy层，服务代理层，透明生成客户端的stub和服务单的skeleton（无论是consumer，还是provider，dubbo都会给你生成代理，代理之间进行网络通信）

第四层：registry层，服务注册层，负责服务的注册与发现（provider注册自己作为一个服务，consumer就可以找注册中心去寻找自己要调用的服务）

第五层：cluster层，集群层，封装多个服务提供者的路由以及负载均衡，将多个实例组合成一个服务（provider可以部署在多台机器上的，多个provider就组成了一个集群）

第六层：monitor层，监控层，对rpc接口的调用次数和调用时间进行监控（consumer调用provider，调用了多少次啊，统计信息监控）

第七层：protocol层，远程调用层，封装rpc调用（负责具体的provider和consumer之间调用接口的时候的网络通信）

第八层：exchange层，信息交换层，封装请求响应模式，同步转异步

第九层：transport层，网络传输层，抽象mina和netty为统一接口

第十层：serialize层，数据序列化层

工作流程：

第一步：provider向注册中心去注册

第二步：consumer从注册中心订阅服务，注册中心会通知consumer注册好服务

第三步：consumer会通过代理走负载均衡策略调用provider的代理，然后调用provider实例服务

第四步：consumer和provider都异步通知监控中心

dubbo的注册中心挂了还可以继续通信：因为开始初始化的时候，消费者会将提供者的地址信息拉取到本地缓存，所以注册中心挂了还是可以继续通信

dubbo都支持哪些通信协议？都支持哪些序列化协议

1.dubbo协议：

默认走的是dubbo协议的，单一长连接，NIO异步通信，基于hessian作为序列化协议

适用场景就是传输数据量很小（每次请求在100Kb以内），但是并发量很高

为支持高并发场景，一般是服务提供者就几台机器，但是服务消费者有上百台，可能每天调用量达到上亿次！此时用长连接是最合适的，就是跟每个服务消费者维持一个长连接就可以，可能总共就100个连接。然后后面直接基于长连接NIO异步通信，可以支撑高并发请求。

否则如果上亿次请求每次都是短连接的话，服务提供者会扛不住的

因为走的是单一长链接，所以传输数据量太大的话，会导致并发能降低，所以建议是传输数据量很小，支撑高并发访问。

2.rmi协议：

走java二进制序列化，多个短连接，适合消费者和提供者数量差不多，适用于文件传输，一般比较少用

3.hessian协议：

走hessian序列化协议，多个短连接，适用于提供者数量比消费者数量还多，适用于文件的传输，一般少用

4.http协议：

走json序列化

5.webservice

走SOAP文本序列化

dubbo支持的序列化协议

dubbo实际基于不同的通信协议，支持hessian、java二进制序列化、json、SOAP文件序列化多种序列化协议。但是hessian是默认的序列化协议

dubbo有哪些负载均衡策略和集群容错策略？ 动态代理策略呢？

1.负载均衡策略

第一种：random loadbalance

默认情况下，dubbo是random load balance随机调用实现负载均衡，可以对provider不同实例设置不同的权重，会按照权重来负载均衡，权重越大分配流量越高，一般就用这个默认的就可以

第二种：roundrobin loadbalance

roundrobin loadbalance，这个默认就是均匀地将流量打到各个服务器上，如果各个机器的性能不一样，容易导致性能差的机器负载过高，所以此时需要调整权重，让性能差的机器承载权重小一些，流量少一些

第三种：leastactive loadbalance

就是自动感知一下，如果某个机器性能越差，那么接收的请求越少，越不活跃，此时会给不活跃的性能差的机器更少的请求

第四种：consistanthash loadbalance

一致性hash算法，相同参数的请求一定分发到一个provider上去，provider挂掉的时候，会基于虚拟节点均匀分配剩余的流量，抖动不会太大。如果需要的不是随机负载均衡，是要一类请求都到一个节点，那就走这个一致性hash策略

2.dubbo集群容错策略

第一种：failover cluster模式

失败自动切换，自动重试其他机器，默认就是这个，常见于读操作

第二种：failfast cluster模式

一次调用失败就立即失败，常见于写操作

第三种：failsafe cluster模式

出现异常时忽略掉，常用于不重要的接口调用，比如记录日志

第四种：failback cluster模式

失败了后台自动记录请求，然后定时重发，比较适合于写消息队列

第五种：forKing cluster模式

并行调用多个provider，只要一个成功就立即返回

第六种：broadcacst cluster

逐个调用所有的provider

3.dubbo动态代理策略

默认使用javassist动态字节码生成，创建代理类

但是可以通过spi扩展机制配置自己的动态代理策略

dubbo的SPI思想是什么？

spi简单来说 Service provider interface

dubbo也是用了spi思想，不过没有用jdk的spi机制，是自己实现的一套spi机制

Protocol protocol = ExtensionLoader.getExtensionLoader（Protocol.class）.getAdaptiveExtension();

Protocol接口，dubbo要判断一下，在系统运行的时候，应该选用这个Protocol接口的哪个实现类来实例化对象来使用呢？

他会去找一个你配置的Protocol，他就会将你配置的Protocol实现类，加载到JVM中来，然后实例化对象，就用你的那个Protocol实现类就可以了

特性是：微内核，可插拔，大量的组件

dubbo里面提供了很多扩展点，如果需要扩展，可以自己写个jar，让consumer或者provider工程，依赖你写的jar，在jar包里指定目录下配置好接口名称对应的文件，里面通过key = 实现类（文件路径）

然后对对应的组件，用类似dubbo:protocol用你的哪个key对应的实现类来实现某个接口，

可以自己去扩展dubbo的各种功能，提供自己的实现

基于dubbo如何做服务治理，服务降级以及重试？

1.服务治理

第一点：调用链路自动生成

服务与服务之间的调用路线

第二点：服务访问压力以及时长统计

其他：服务分层（避免循环依赖），调用链路失败监控和报警，服务鉴权，每个服务的可用性的监控

2.服务降级

现在就是基于mock，如果调用失败统一返回null

但是可以将mock设置为true，然后在跟接口同一个路径下实现一个Mock类，命名规则是接口名称加Mock后缀，然后在Mock类里实现自己的降级逻辑

3.失败重试和超时重试

所谓失败重试，就是consumer调用provider要是失败了，比如有异常，此时应该重试的，或者调用超时了也可以重试

分布式服务接口的幂等性如何设计？

这个不是技术问题，这个没有通用的一个方法，是一个需要结合保证幂等性

保证幂等性主要是三点：

1.对于每个请求必须有一个唯一的标识，举例子：订单支付请求，肯定包含订单id，一个订单id最多支付一次

2.每次处理完请求之后，必须有一个记录标识这个请求处理过了，比如常见的方案是在mysql中记录数据状态，比如支付之前记录一条该订单的支付流水，而且支付流水采用orderld作为唯一键，只有成功插入这个支付流水，才可以执行实际的支付扣款

3.每次接收请求需要进行判断之前是否处理过的逻辑处理，比如说，如果有一个订单已经支付了，就已经有了一条支付流水，那么如果重复发送这个请求，则此时先插入支付流水，orderld已经存在了，唯一键约束生效，报错插入不进去的。就可以不再扣款

如何保证分布式系统的接口调用顺序性？

1.通过hash分发到不同服务器上不同的请求，然后将请求存入内存队列中，让线程从内存队列中获取有序请求，并进行数据库操作

2.基于zookeeper做分布式锁，客户端请求，将请求放入mq中，然后通知服务器获取请求，然后通过zookeeper加锁判断请求顺序，再释放锁，让剩下请求进来执行操作

如何设计一个类似dubbo的rpc框架？

简单设计思路：

1.服务需要注册中心，可以基于zookeeper做注册中心

2.还需要做provider的一个动态代理层

3.消费者通过代理层去注册中心获取信息的过程，而每个provider的服务可能在多台机器上

4.然后发送到哪台服务器上，需要一个负载均衡算法进行分配

5.找到服务器后，要考虑如何发送请求，可以使用netty，NIO方式；可以发送请求后，我们需要考虑发送什么样格式数据，可以使用hessian序列化协议

6.请求过来，要监听某个网络端口，然后做一个动态代理调用你的本地服务
