MySQL框架探索I

MySQL驱动

MySQL驱动就是在底层跟数据库建立网络连接

数据库连接池

为避免多个线程反复循环创建连接，销毁连接的情况，会使用数据库连接池，来维护多个数据库连接，不销这些数据库连接，而是放回数据库连接池中，等待其他继续使用

从数据库连接池中获取SQL后执行流程

1.网络连接后必须让线程来处理（由线程去监听和读取请求）

2.线程把SQL语句交给SQL接口执行

3.SQL接口解析SQL后交给SQL解析器

4.SQL解析器把解析好的SQL交给查询优化器（选择最优查询路线）

5.交给存储引擎执行SQL语句

6.存储引擎去内存或者磁盘获取数据

InnoDB架构设计，如何基于存储引擎完成一条更新SQL执行

InnoDB重要内存结构：缓冲池

引擎在执行更新语句的时候，会先看一下缓冲池，如果没有，那么会直接从磁盘里加载到缓冲池里来，并对这行记录加独占锁

undo日志：更新的数据进行回滚

存储引擎考虑数据回滚需要，会把更新前的值写入undo日志文件中

更新buffer pool中的缓存数据

开始更新操作的时候，先是更新缓冲池中的记录，此时这个数据就是脏数据

Redo Log Buffer：万一系统宕机，如何避免数据丢失？

为避免执行写操作时候宕机，就必须把对内存所做的修改写入到一个Redo Log Buffer里，也是一个内存里的一个缓冲区，用来存放redo日志

就算此时写操作没提交事务宕机了，也没关系，丢失的也只是缓冲数据，磁盘数据还是原来的样子，后续可以执行补救方案

提交事务的时候将redo日志写入磁盘

当提交一个事务，会根据一定的策略把redo日志从redo log buffer里刷入磁盘文件里

策略是通过InnoDB_flush_log_at_trx_commit来配置

参数值为0时，不会把redo log buffer里的数据刷入磁盘文件，异常情况会丢失内存的全部数据

参数值为1时，提交事务时，必须把redo log从内存刷入到磁盘文件里，只要事务提交成功，那么redo log就必然在磁盘里

参数值为2时，提交事务，把redo日志写入磁盘文件对应的os cache缓存里，而不是直接进入磁盘文件，出现异常情况也会丢失os cache数据，让你感觉事务已经提交了，但是并没有

MySQL binlog到底是什么

bin log叫做归档日志，不是InnoDB存储引擎特有的日志，是MySQL server自己的日志

在提交事务的时候，会把redo log日志写入磁盘文件中，还会把这些操作更新到对应的binlog日志文件中

binlog日志的刷盘策略分析

对于binlog也有不同刷盘策略，通过一个sync_binlog参数控制binlog刷盘策略

参数值为0时，binlo写入磁盘时，不是直接进入磁盘文件，而是进入os cache内存缓存

参数值为1时，会强制在提交事务的时候，把binlog直接写入到磁盘文件里

基于binlog和redo log完成事务的提交

把binlog写入磁盘文件之后，接着完成最终的事务提交，把本次更新对应的binlog文件和这次更新的binlog日志在文件里的位置，都写入到redo log日志文件里，同时在redo log日志文件里写入一个commit标记

commit标记的意义：用来保持redo log日志与binlog日志一致的

如何配置Buffer Pool的大小

默认是128M,如果是16核32G的机器，可以分配个2G

设置Innodb_buffer_pool_size = 2147483648就可以

磁盘上的数据页和Buffer Pool中的缓存页如何对应上的

实际上磁盘中存放的数据页大小是16kb，而buffer pool与数据页是对应的，也是16kb

缓存页对应存放的信息

把数据页加载到buffer pool里去，会存放对应的数据页所属的表空间，数据页的编号，还有buffer pool的地址

如何知道哪些缓存页是空闲的

数据库会为Buffer pool设计一个free链表（双向链表），在free链表里，每个节点就是一个空闲的缓存页描述数据块的地址，也就是，只要一个缓存页是空闲的，那么他的描述数据块就会被放入到free链表中

如何知道哪些缓存页是脏页

最终内存更新的脏页数据是要被刷新回磁盘文件的，所以数据库这里会引入一个跟free链表类型的flush链表。

flush链表本质也是通过缓存页的描述数据块中的两个指针，让修改过的缓存页的描述数据块，组成一个双向链表

对于缓存页要是满了，该如何淘汰更新缓存页

就通过缓存页的缓存命中率来判断，缓存命中率低的就刷新到磁盘文件中，那如何判断哪些缓存页经常被访问，或者很少被访问呢，数据库又引入了一个LRU链表，对于经常使用到的缓存页会被放入到链表头部

MySQL的预读机制会给LRU带来隐患

通过预读机制，会将一些数据直接被放入到LRU链表头部，导致一些热点访问数据被挤到尾部

那么什么情况下会触发MySQL预读机制呢？

1.参数配置InnoDB_read_ahead_threahold，默认是OFF关闭状态的，默认值为56，意思是如果顺序访问一个区里的多少个数据页，访问的数据页数量超过了参数值，会触发预读机制

2.如果buffer pool里缓存一个区里的13个连续的数据页，而且这些数据页都是比较频繁被访问的，此时会直接触发预读机制，把这个区里的其他数据页都加载到缓存里。

另外一种可能会导致频繁被访问的缓存页被淘汰的场景，就是全表扫描

一下会将表里查出所有的数据页都从磁盘加载到buffer pool里，然后装入各个缓存页里

基于冷热分离的思想设计LRU链表

真正LRU链表的设计的是基于冷热分离思想做的，把LRU拆分成两个部分，热数据和冷数据，占比是由参数InnoDB_old_blocks_pct控制，默认是37，就是冷数据占比37%

MySQL事务隔离级别

1.read uncommitted（读未提交）

2.read committed（读已提交）

3.repeatable read（可重复读）

4.serializable（串行化）
